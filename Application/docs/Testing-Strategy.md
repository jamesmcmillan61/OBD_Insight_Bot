# Testing Strategy – IBM OBD InsightBot

This document outlines the **final testing approach**, standards, and quality assurance processes used for the **IBM OBD InsightBot** project. It reflects the **completed implementation**, **automated CI/CD pipelines**, **executed test suites**, and **actual measured code coverage**.

**Status**: Project Complete  
**Project Phase**: Final Delivery  
**Last Updated**: April 2026  
**Maintained By**: IBM OBD InsightBot Team  

---

## Testing Philosophy

The testing strategy for IBM OBD InsightBot is based on the following core principles:

1. **Quality First**  
   Testing is embedded throughout development and deployment rather than treated as a final step.

2. **Evidence‑Driven Validation**  
   Test success, deployment health checks, and coverage reports provide measurable evidence of quality.

3. **Automation and Consistency**  
   All automated tests run via Azure DevOps pipelines to ensure consistency across environments.

---

## Test Coverage Requirements

Testing and coverage are guided by RNF‑06 in the Software Requirements Specification. While initial targets were defined during planning, final coverage reflects real implementation constraints and deliberate exclusions.

### Coverage Rules

- **Measured Coverage**: Actual coverage measured via pipeline execution
- **Excluded from Coverage Calculations**:
  - Auto‑generated Razor views
  - EF Core migration files
  - Startup and boilerplate code
  - UI markup files (`.cshtml`, generated view classes)

## Test Types

### 1. Unit Tests

**Purpose**  
Validate individual methods, classes, and services in isolation.

**Scope**:
- Controllers
- Service layer logic
- Validation helpers
- Domain models

**Frameworks**:
- ASP.NET Core: xUnit
- Python Backend: pytest

---

### 2. Integration Tests

**Purpose**  
Validate interactions between components and subsystems.

**Scope**:
- API request/response handling
- Database persistence via SQLite
- AI service orchestration
- Session management workflows

Integration tests are executed automatically in the CI pipeline prior to deployment.

---

### 3. Functional Testing

**Purpose**  
Verify the system satisfies all documented **Functional Requirements (FRs)**.

**Functional Coverage**:
- Upload OBD‑II diagnostic data
- Parse and analyse fault codes
- Generate AI explanations
- Maintain conversational context
- Persist and retrieve user sessions

Functional testing is achieved through controller tests, integration tests, and system‑level validation.

---

### 4. Acceptance Testing

**Purpose**  
Confirm the system is ready for delivery and real‑world use.

Acceptance testing is implemented automatically as part of **CI/CD deployment pipelines**.

#### Acceptance Criteria

The system is accepted when:

-  All tests pass
-  Coverage reports are produced
-  Backend API starts successfully
-  `/health` endpoint returns HTTP 200
-  Web application deploys and runs in IIS
-  No critical runtime errors occur

Pipeline success constitutes formal acceptance evidence.

---

### 5. System and Deployment Testing

#### Backend API

- Deployed as a Windows service using NSSM
- Automatic startup validation
- Extended health‑check polling to allow AI model load time
- Deployment fails if health checks do not pass

#### Web Application

- Deployed to IIS
- App pool lifecycle validation
- SQLite database preserved across deployments
- Post‑deployment HTTP health checks
